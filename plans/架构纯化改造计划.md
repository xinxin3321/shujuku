# 架构纯化改造计划

> 前置文档：`docs/架构违规报告.md`（84 条违规 import）  
> 目标：消除全部违规，达到严格单向 `presentation → service → data → shared`  
> 分支：从 `001-three-layer-refactor` 拉出 `002-arch-purify`  
> 约束：每个阶段结束必须 `npm run build` 通过 + 功能等价

---

## 改造原则（从违规报告第六节搬过来，执行时对照）

```
presentation  只做 UI 编排，调 service 拿结果后自己更新 DOM
service       纯逻辑 + 返回值，绝不 import presentation
data          纯 CRUD + 参数化，绝不 import service / presentation
shared        无状态工具/常量/类型/宿主 API 引用，被所有层引用
```

## 阶段 P3：service 层纯化 — 消除全部 service → presentation 依赖

> **核心原则：service 层绝不主动调 UI。不是"怎么让 service 间接调 UI"，而是"这段代码到底该放在哪一层"。**

| 类型 | 含义 | 文件 | 违规数 |

| **A类** | 整个文件本身就是编排/UI入口，应搬到 presentation 层 | api-registry.ts, init.ts, admin.ts | 17 |
| **B类** | 文件核心是纯业务，但混入了少量 UI 副作用行，需提取这几行到 presentation 调用方 | helpers-remaining, update-process, import-process, content-optimization, settings-service, merge-logic, pipeline, injection-engine | 14 |

```
 └→ P1 data 层纯化（14 条 → 0）
     └→ P2 错放函数归位（presentation 内纯逻辑 → service）
         └→ P3 service 层纯化（48 条 → 0）
             └→ P4 presentation → data 清理（36 条 → 0）
                 └→ P5 收尾 + 护栏
```

严格串行。每阶段完成必须：
1. `npm run build` 通过
2. `check-arch.sh` 违规数 ≤ 预期
3. 关键路径手动测试通过

## 预期工作量估算

| 阶段 | 涉及文件数 | 核心难点 | 预估 |
|------|-----------|----------|------|
| P0 | 4 个新文件 + state-manager 拆分 | re-export 兼容，不破坏现有 import | 中 |
| P1 | ~10 个 data 文件 + 调用方更新 | table-repo 拆分，参数化改造 | 中 |
| P2 | ~5 个文件迁移 | 函数依赖链理清 | 中 |
| P3 | 12 个 service 文件 | 最大最难：每个函数都要从"有副作用"改为"纯返回值" | 高 |
| P4 | ~12 个 presentation 文件 | 数量多但模式固定：加 service 中转层 | 中 |
| P5 | 清理 + 测试 | 护栏脚本 + 全量验证 | 低 |

---

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| P0 state-manager 拆分后大面积 import 失败 | 先保留 state-manager 做 re-export 门面，逐步迁移引用 |
| P3 service 函数改为返回值后，presentation 调用方需要大量改写 | 逐个 service 文件改，每改一个验证一次 |
| api-registry 是 window API 的路由层，改造影响外部调用者 | 保持 `window.AutoCardUpdaterAPI` 的接口签名不变，只改内部实现 |
| 改造过程中功能回归 | 每阶段手动测试关键路径 |
