# 修复表格模板导入后预设列表与切换异常

## 已确认事实

### 1. 模板预设下拉的选项来源与刷新方式
- `src/presentation/components/template-preset-ui.ts`
  - `loadTemplatePresetSelect_ACU()` 会重建“全局模板预设”和“当前聊天模板预设”两个下拉框。
  - 全局预设选项来源于 `listTemplatePresetNames_ACU()`。
  - 当前聊天下拉除了全局预设名，还会叠加当前聊天本地快照项。
  - `refreshPresetUIAfterSwitch_ACU()` 最终也是调用这里的刷新逻辑。

结论：**下拉“不刷新”不是因为没有重建 DOM，而更像是刷新时使用了已经被错误改写的状态。**

### 2. 全局导入入口会直接应用模板，而不只是保存到预设库
- `src/presentation/triggers/data-admin-ui.ts`
  - `importTableTemplate_ACU({ scope: 'global' })` 在读取并解析模板后：
    1. 先 `upsertTemplatePreset_ACU(derivedPresetName, prepared.templateStr)` 保存到全局预设库；
    2. 再调用 `applyTemplateSnapshotToScope_ACU(prepared.templateStr, { scope: 'global', source: 'ui_global_import', presetName: derivedPresetName, save: true, persistChatScope: false })`。
- `src/presentation/bootstrap/api-groups/template-preset-api.ts`
  - `importTemplateFromData()` 对 global import 也是同样流程：
    1. 先保存预设；
    2. 再调用 `applyTemplateSnapshotToScope_ACU(...)`。

结论：**当前系统把“导入到全局模板库”和“切换当前全局模板”做成了同一件事。**

### 3. applyTemplateSnapshotToScope_ACU 的副作用比名字大得多
- `src/service/template/template-preset-service.ts`
  - `applyTemplateSnapshotToScope_ACU()` 会：
    1. `_set_TABLE_TEMPLATE_ACU(snapshot.templateStr)` —— 立即改当前运行态模板；
    2. 如果 `scope === 'global'`，执行 `saveCurrentProfileTemplate_ACU(...)`；
    3. 调用 `persistTemplateScopeSelectionState_ACU(...)` —— 写入全局预设名或聊天作用域状态；
    4. 调用 `applyTemplateScopeForCurrentChat_ACU()` —— 重新按聊天作用域优先级覆盖当前运行态；
    5. 刷新 merged data。

结论：**这个函数不是“应用快照”这么简单，它会同时修改运行态、持久化状态、聊天作用域与后续解析结果。**

### 4. 当前聊天作用域优先级高于全局模板
- `src/service/settings/settings-service.ts`
  - `applyTemplateScopeForCurrentChat_ACU()` 的优先级是：
    1. `chat_override` 的 `templateStr`
    2. `preset_link` 对应的全局预设
    3. 当前全局模板快照
- `src/service/template/chat-scope/chat-scope-template.ts`
  - `activateChatTemplatePresetSelection_ACU()` 在切换当前聊天预设时，若命中本地快照则写成 `chat_override`；否则写成 `preset_link`。

结论：**一旦当前聊天已经进入 `chat_override`，之后切换全局预设不会改变当前聊天实际模板。用户感知成“切不了”，本质是聊天级快照继续压住全局。**

### 5. 全局导入按钮本身没有额外偷偷写聊天作用域
- `src/presentation/pages/popup-bindings-data.ts:434-437`
  - “导入模板”按钮只是调用 `importTableTemplate_ACU({ scope: 'global' })`。

结论：**UI 绑定层不是主因，主因在导入实现链。**

## 问题判断

### 根因判断
这次异常不是单一 UI 刷新 bug，而是以下两层问题叠加：

1. **语义耦合错误**
   - “导入全局模板到预设库”和“切换当前全局模板”被绑死。
   - 导入新模板后，当前全局预设会被强制改成新导入项。

2. **作用域优先级造成的表面锁死**
   - 当前聊天如果持有 `chat_override`，后续再切全局模板不会生效。
   - UI 刷新只是把这种高优先级状态重新显示出来，看起来像导入后“无法切换”。

### 对用户症状的映射
- “导入新的表格模板后下拉菜单不会刷新”
  - 更准确地说：**刷新后选中状态与实际作用域状态已经被导入流程改写，因此表现为列表/选项状态异常，而不是单纯没刷新。**
- “当前表格预设也会被强制切换成刚刚导入的模板预设”
  - 这是已确认事实，由 global import 直接调用 `applyTemplateSnapshotToScope_ACU(...scope:'global'...)` 导致。
- “且无法切换成其它的，必须再次重启窗口后才能正常切换”
  - 高概率是当前聊天已经处于 `chat_override` 或相关运行态被导入逻辑立即覆盖，后续 UI 再切换时被作用域优先级压住；重启后重新装载状态，现象暂时缓解。

## 实施方案

### 方案目标
把三件事彻底拆开：
1. 导入到全局预设库
2. 刷新 UI 列表
3. 是否切换当前全局/当前聊天实际模板

默认情况下，**导入全局模板只新增/覆盖全局预设项，不自动改当前生效模板。**

### 具体实施步骤

#### Step 1：拆分 global import 的保存与应用语义
涉及文件：
- `src/presentation/triggers/data-admin-ui.ts`
- `src/presentation/bootstrap/api-groups/template-preset-api.ts`
- `src/service/template/template-preset-service.ts`

实施内容：
- 为 global import 引入“仅保存预设并返回结果”的路径，不再直接复用 `applyTemplateSnapshotToScope_ACU()`。
- `chat` import 继续保留当前行为，因为聊天导入本来就应生成聊天级模板快照/作用域。
- 具体可以采用以下任一最终实现方式（实现时二选一，但都必须遵守同一语义边界）：
  - 新增 service 函数，例如 `importTemplatePresetToLibrary_ACU()`，负责解析后的全局预设写入；
  - 或保留现有存储函数组合，在 UI/API 层对 global import 只执行 `upsertTemplatePreset_ACU()` + UI 刷新，不调用 `applyTemplateSnapshotToScope_ACU()`。

验收点：
- 全局导入完成后，当前运行态模板和 `settings_ACU.currentTemplatePresetName` 不应被隐式切换。

#### Step 2：保留聊天作用域优先级，但避免被 global import 污染
涉及文件：
- `src/service/settings/settings-service.ts`
- `src/service/template/chat-scope/chat-scope-template.ts`
- `src/service/template/template-preset-service.ts`

实施内容：
- 不改 `chat_override > preset_link > global` 这个优先级规则，这条规则本身没问题。
- 重点检查 global import 过程中是否有任何路径会额外写入聊天状态、重建聊天快照、或触发错误的 presetName 关联。
- 若实现时发现 global import 后仍会连带影响聊天状态，则补充保护：
  - global import 不得调用任何会写 `chat_override` / `preset_link` 的聊天级持久化路径；
  - global import 结束后只刷新 UI，不应触发聊天模板重绑定。

验收点：
- 当前聊天原本是 `chat_override` 时，导入全局模板后它仍保持原聊天快照；
- 当前聊天原本跟随全局时，导入全局模板后也不应自动跳到新模板，除非用户手动切换。

#### Step 3：修正导入后的 UI 刷新与回填策略
涉及文件：
- `src/presentation/components/template-preset-ui.ts`
- `src/presentation/components/pipeline-ui-helpers.ts`
- `src/presentation/pages/popup-bindings-data.ts`

实施内容：
- global import 成功后必须立即刷新模板预设下拉，让新导入项立刻出现在列表中。
- 刷新时：
  - 全局下拉按“当前真实全局预设”回填，而不是强制选中新导入名；
  - 聊天下拉按“当前真实聊天作用域”回填，而不是沿用过期 DOM 里的值。
- 检查 `keepGlobalValue` / `globalSelectName` 的使用点，避免把“仅用于刷新视觉选中项”的参数错误地当成“切换状态”的替代。
- 必要时补强状态文案，让用户能分辨：
  - 当前聊天正在跟随全局；
  - 当前聊天正在使用本地快照，因此切全局不会影响它。

验收点：
- 导入新模板后，无需关闭窗口即可在下拉中看到它；
- 导入后仍能正常切换到其它预设；
- 若当前聊天被本地快照锁定，UI 文案必须明确反映这一点。

#### Step 4：检查所有调用点，统一语义
涉及文件：
- `src/presentation/pages/popup-bindings-data.ts`
- `src/presentation/bootstrap/api-groups/template-preset-api.ts`
- `src/presentation/triggers/data-admin-ui.ts`

实施内容：
- 逐一检查以下入口是否仍错误复用“导入即应用”的路径：
  - 全局导入按钮
  - API 导入
  - 聊天导入
  - 全局保存 / 另存为
  - 聊天保存到全局
- 目标是让不同入口遵循一致规则：
  - **导入到全局库**：只改预设库
  - **切换全局预设**：只改全局当前选择
  - **切换聊天预设**：只改聊天作用域
  - **聊天导入**：允许生成聊天级快照

验收点：
- 不再存在某个隐藏入口仍然会把 global import 变成 forced switch。

#### Step 5：补验证
实施完成后至少验证以下场景：

1. **全局导入新模板**
   - 导入后下拉立即出现新模板名；
   - 当前全局预设不自动跳转；
   - 当前聊天实际模板不被意外修改。

2. **全局导入后切换其他预设**
   - 可直接切换到任意已有全局预设；
   - 无需重启窗口。

3. **当前聊天是 chat_override**
   - 导入全局模板后，聊天仍保持自身模板；
   - UI 明确显示聊天级快照仍在生效；
   - 聊天下拉仍可切换其它聊天/链接预设。

4. **当前聊天是 preset_link / inherit_global**
   - 导入全局模板后不自动切走；
   - 手动切换后立即生效。

5. **双入口一致性**
   - 从 UI 导入与从 API 导入表现一致。

## 风险与注意项

- `applyTemplateSnapshotToScope_ACU()` 当前职责过重。如果只是局部绕开 global import 对它的复用，能先止血，但函数语义仍然偏混乱。实现阶段如果改动可控，建议顺手把命名或职责再收紧一点，否则以后还会有人再次误用。
- 不应通过“导入成功后再强行 refresh 一遍 select 值”来掩盖问题。那只是给错误状态刷一层新漆，过一会儿还是会掉。
- 不建议改动 `applyTemplateScopeForCurrentChat_ACU()` 的优先级规则。用户当前抱怨的不是这条规则本身，而是 global import 不该碰到这条规则。

## 最终建议

推荐按“**先拆 global import 语义，再修 UI 回填**”的顺序实施。先碰 UI 不碰状态，只会把 bug 从“稳定复现”改成“偶发错乱”，那种修法质量太差，不配进主干。
