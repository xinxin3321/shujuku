# SQL 模板输入规范化与自动纠错实施计划

## 1. 目标

为 SQLite 模式补齐一层可靠的输入规范化机制，解决类似以下两类问题：

1. **DDL / SQL 语法字符污染**
   - 例如模板里写出 `CHECK(quantity >＝ 0)`，其中 `＝` 为全角字符，导致 SQLite 解析失败并报 `near "0": syntax error`
2. **受约束字段值格式不规范**
   - 例如 `code_index` 约束要求 `AM[0-9][0-9][0-9][0-9]`，但运行时写入了大小写、全半角、空白不规范的值，触发 `CHECK constraint failed`

本次优化的本质目标不是“修一个模板”，而是让 **模板导入链路 + 运行时 SQL 执行链路** 都具备可控的自动兼容能力，并对无法安全自动修复的情况保留明确失败。

## 3. 设计原则

### 3.1 必须自动修复的内容

仅对**可安全判断语义**的内容自动转换：

1. **SQL 结构字符兼容化**
   - 全角运算符 / 标点 / 括号在 SQL 结构位置上转换为 ASCII
   - 目标是让 SQLite 能正确解析 DDL / DML
2. **白名单字段的值 canonicalization**
   - 只对有明确约束模式的字段做自动规范化
   - 当前首个明确目标字段：`code_index`
   - 例如：trim、NFKC、字母转大写、全角数字转半角数字

### 3.2 不能静默乱改的内容

以下情况不能用“自动修复”掩盖：

1. 值经过规范化后仍不满足约束
   - 如 `AM12`、`AX0001`、`AM00001`
   - 这类必须继续失败，并保留明确错误
2. 普通正文文本
   - 如 `chronicle_text`、`summary`、对话、描述文本中的全角标点通常是合法内容
   - 不能为了兼容 SQL 结构把正文内容一起篡改
3. 无法确定语义的通用 TEXT 列
   - 不能对所有字符串列无差别大写、trim、替换字符

### 3.3 规范化必须分层，不可散落

不能在多个调用点各写一堆 `replace()`。正确做法是：

- 提供统一 sanitizer / normalizer 模块
- 明确区分：
  - SQL 结构规范化
  - 受约束值规范化
- 由模板导入入口和运行时写入入口复用该模块

## 4. 推荐实现方案

### 4.1 新增共享规范化模块

建议新增一个共享模块，位置优先考虑：

- `src/data/sqlite/sql-normalizer.ts`
  或
- `src/shared/sql-normalizer.ts`

最终位置以项目现有分层约定为准，但职责必须集中。

该模块至少提供以下能力：

#### A. SQL 结构兼容化

输入：原始 SQL 文本

输出：结构字符已兼容化的 SQL 文本

处理内容建议包括：

- 全角等号 `＝` -> `=`
- 全角大于号 `＞` -> `>`
- 全角小于号 `＜` -> `<`
- 全角括号 `（` `）` -> `(` `)`
- 全角逗号 `，` -> `,`
- 其他明确会影响 SQLite 结构解析的兼容字符

关键要求：

- 优先只处理 **SQL 结构位置**
- 若实现成本过高，可先采用更保守策略：
  - 仅在 DDL / DML 非字符串字面量区段进行替换
- 必须避免误改字符串字面量中的正文内容

#### B. 受约束字段值 canonicalization

建议提供按列名白名单触发的值规范化函数，例如：

- `normalizeConstrainedValue(columnName, rawValue)`

首批明确规则：

- `code_index`
  - `trim()`
  - Unicode NFKC 兼容归一化
  - 英文字母转大写
  - 全角数字转半角数字

输出后再检查是否满足目标模式；若仍不满足，则不吞错。

## 5. 接入点设计

### 5.1 模板导入链路接入

涉及文件：

- `src/data/sqlite/schema-mapper.ts`
- `src/data/sqlite/sync-bridge.ts`

目标：

1. 建表前先规范化 DDL
2. seed 数据生成 INSERT 时，对白名单受约束字段做值规范化

建议落点：

#### 方案首选

在 `schema-mapper.ts` 内部接入：

- `generateDDL()` 返回前对 `sheet.sourceData.ddl` 做 SQL 结构兼容化
- `generateInserts()` 在 `escapeValue()` 之前按列名执行值规范化

优点：

- 规范化与 SQL 生成职责靠近
- `SyncBridge` 保持“桥接层”定位，不被塞入过多细节

#### SyncBridge 配合职责

`sync-bridge.ts` 只保留：

- 失败时更清晰的日志
- 若发生自动修复，可补充 debug / warn 日志，提示该 sheet 的 DDL 或 seed 数据已被规范化处理

### 5.2 运行时 SQL 写入链路接入

涉及文件：

- `src/service/table/sql-table-service.ts`

目标：

在 AI 生成 SQL 真正执行前，统一走一次规范化。

建议落点：

- `applyEdits()`：`splitSqlStatements(cleaned)` 之后、`engine.runBatch(statements)` 之前
- `executeMutation()`：`engine.run(sql, params)` 之前

建议行为：

1. 对整句 SQL 做结构兼容化
2. 对可识别的 INSERT / UPDATE 中白名单字段执行值 canonicalization
3. 若发生过自动修正，可记录 debug/warn 日志，便于排查 AI 输出质量问题

注意：

- `executeQuery()` 不应做会改变语义的写入型规范化
- 只读查询保持原样，避免调试认知混乱

## 6. 边界与风险控制

### 6.1 结构字符替换风险

风险：如果直接对整条 SQL 做全局替换，可能误改字符串字面量中的合法正文。

控制策略：

1. 优先实现“跳过字符串字面量”的扫描式替换
2. 若首版只覆盖 DDL / 简单 DML，也必须明确处理边界
3. 不允许为了省事把所有文本一起 replace

### 6.2 值规范化越权风险

风险：如果对所有 TEXT 值都做 trim / upper / NFKC，会把合法业务文本改坏。

控制策略：

1. 只对白名单列生效
2. 首批只做 `code_index`
3. 后续扩展必须以真实约束为依据，不允许拍脑袋加规则

### 6.3 自动修复掩盖脏数据风险

风险：如果规范化失败后仍继续执行，会把数据问题隐藏掉。

控制策略：

1. 规范化后必须再次验证目标模式
2. 不通过则继续抛错
3. 错误日志中要能看出是“原始值不合法”还是“规范化后仍不合法”

## 7. 测试计划

### 7.1 需要补充/修改的测试范围

已确认相关测试文件至少包括：

- `tests/service/table/sql-table-service.test.ts`

此外建议新增更聚焦的测试文件，候选位置：

- `tests/data/sqlite/schema-mapper.test.ts`（若当前没有，则新增）
- 或在现有 table/sqlite 相关测试中补充用例

### 7.2 必须覆盖的测试用例

#### A. DDL 结构字符兼容化

1. 含全角等号的 DDL 能自动修复并成功建表
   - 例如：`CHECK(quantity >＝ 0)`
2. 含全角括号、逗号等结构字符的 DDL 能正确规范化

#### B. seed INSERT 值规范化

1. 模板数据中的 `code_index='am0001'` 最终能按 `AM0001` 写入
2. 模板数据中的 `code_index='ＡＭ０００１'` 能正确规范化

#### C. 运行时 SQL 写入规范化

1. `applyEdits()` 执行 INSERT 时，`code_index='am0002'` 自动规范化为 `AM0002`
2. `executeMutation()` 执行 UPDATE 时，同样适用

#### D. 非法值仍应失败

1. `code_index='AM12'` 规范化后仍非法，必须报错
2. `code_index='AX0001'` 规范化后仍非法，必须报错

#### E. 正文内容不被误改

1. `chronicle_text` / `description` 等正文中的全角标点保留原样
2. 只修复 SQL 结构，不篡改正文文本

## 9. 验收标准

满足以下条件才算完成：

1. 使用类似 `@YO改-骰子-SQL版v3.json` 的模板时，`sheet_inventory` 不再因 `>＝` 这类兼容字符报 `near "0": syntax error`
2. `code_index` 常见格式问题（大小写、全角字母数字、首尾空白）可自动规范化
3. 规范化后仍不合法的值继续明确报错，而不是静默吞掉
4. 正文文本字段不会被结构规范化逻辑误改
5. 相关测试覆盖导入链路与运行时写入链路

## 10. 明确不做的事

以下内容本次不应顺手扩张：

1. 不做“所有枚举值自动猜测修复”
   - 没有明确规则就不要乱补
2. 不做“任意 SQL 容错解析器”
   - 目标是兼容常见、可安全归一的输入问题，不是重写 SQLite 解析器
3. 不把所有报错都转成自动修复
   - 无法确定语义时，保留失败比静默篡改安全得多

---

## 11. 质量判断

这版计划已经基于真实代码路径与真实模板样本，不再是空泛口号。但也要说清楚：

- 计划已经足够进入实现
- 真正实现时最容易写烂的地方，是“为了省事直接全局 replace 整条 SQL”
- 如果实现阶段这么干，正文内容迟早会被误伤，那就不是优化，是制造新事故

因此实现时的第一优先级不是“尽快修好报错”，而是**把可自动修复的边界定义清楚**。
