# 解决打包后反复出现 TavernStorage 根对象不可用告警（仅 JS 版，不能影响插件版）

## 目标

消除运行期反复出现的日志：

```text
WARN TavernStorage [shujuku_v120] [TavernStorage] 酒馆设置根对象不可用, 返回 null
```

**新增且关键的约束：这是打包后的 JS 版专属错误，插件版没有这个问题。**

因此本次修复目标不是“统一重构所有运行模式”，而是：

1. 只修复 userscript / 打包后单文件 JS 版的初始化与降级问题；
2. 明确保护插件版现有行为，不主动修改插件版入口逻辑；
3. 避免把共享模块的修复写成全局行为，误伤正常的 extension 路径。

## 已确认事实

### 1. 告警定义点

- 文件：`src/data/storage/tavern-storage.ts`
- 函数：`getTavernSettingsNamespace_ACU()`
- 行为：调用 `tryReadBridgeFromTop_ACU()` 后，如果 `tavernExtensionSettingsRoot_ACU` 仍为空，则打印：
  - `"[TavernStorage] 酒馆设置根对象不可用, 返回 null"`
- 然后返回 `null`。

### 2. 重复触发根因已经定位到调用链，不是孤立日志问题

- `getConfigStorage_ACU()` 每次调用都会先执行 `getTavernSettingsNamespace_ACU()`。
- 这意味着所有配置读取/写入路径都会先探测 Tavern root。
- `migrateKeyToTavernStorageIfNeeded_ACU()` 内部再次调用 `getConfigStorage_ACU()`。

### 3. `loadSettings_ACU()` 当前顺序会主动制造竞态

- 文件：`src/service/settings/settings-service.ts`
- 当前流程：
  1. `void initTavernSettingsBridge_ACU()` —— 异步启动，但**不等待完成**
  2. 立刻调用两次 `migrateKeyToTavernStorageIfNeeded_ACU(...)`
  3. 立刻调用 `getConfigStorage_ACU()` 读设置
- 结果：桥接未完成时，单次设置加载链路就可能连续触发多次相同 warn。

### 4. 高频读取方不止一个

- `src/data/repositories/profile-repo.ts` 中：
  - `loadGlobalMeta_ACU()`
  - `saveGlobalMeta_ACU()`
  - `readProfileSettingsFromStorage_ACU()`
  - `readProfileTemplateFromStorage_ACU()`
  - `writeProfileSettingsToStorage_ACU()`
  - `writeProfileTemplateToStorage_ACU()`
- 这些函数都直接走 `getConfigStorage_ACU()`。
- 所以只修一个入口而不修存储门面，仍会重复告警。

### 5. userscript 与插件的入口路径不同，不能混为一谈

#### userscript / JS 版入口

- 文件：`src/index.ts`
- 行为：document ready 后直接调用 `mainInitialize_ACU()`
- 这里没有插件版那种就绪等待逻辑
- 这条路径更容易出现“bridge 尚未可用，但设置读取已经开始”的竞态

#### 插件版入口

- 文件：`src/entry-extension.ts`
- 行为：先执行 `waitForTavernHelper()`，通过后才调用 `mainInitialize_ACU()`
- 用户已明确说明：**插件版没有这个问题**
- 因此插件入口不是本次修复主战场，而是保护对象

### 6. 运行时环境分流点已经明确

- 文件：`src/shared/runtime-env.ts`
- 插件入口通过 `_forceExtensionMode()` 强制设为 extension
- userscript 模式依赖 `window.parent !== window` 检测为 iframe 环境
- 这意味着可以在共享模块中按 `isExtensionMode()` 精确分流行为

### 7. 构建模式事实

- 默认 `npm run build` 产物是 userscript：`dist/index.bundle.js`
- `npm run build:extension` 产物才是插件：`dist/extension/index.js`
- 用户现在已明确：问题属于“打包后的 JS 版”，**不是插件版**
- 所以后续验证必须以 userscript / 单文件 JS 场景为主，并额外做插件回归确认

## 根因判断

根因不是“插件初始化普遍有问题”，而是：

> **userscript / 打包后 JS 版中，Tavern bridge 的可用时机晚于同步配置读取时机，导致 `getConfigStorage_ACU()` 在启动窗口内反复命中空 root，并把同一个问题放大成多次 warn。**

这个问题被以下机制共同放大：

1. `loadSettings_ACU()` 不等待 bridge 初始化完成；
2. `getConfigStorage_ACU()` 作为统一门面，每次都立即探测 Tavern root；
3. profile/globalMeta/template 等多个同步读取方都会反复走该门面；
4. userscript 入口 `src/index.ts` 在 document ready 后直接启动，没有插件入口那套等待保护；
5. 存储层没有“初始化中 / 已降级 / 已报告”的状态缓存与单次告警机制。

**插件版当前正常，因此不能把“修 userscript 竞态”误写成“重构插件入口”。** 那样改动方向本身就是错的。

## 修复原则

1. **修复范围限定在 userscript / 非 extension 路径**
   - 共享模块可以改，但必须通过环境分支限制影响面
   - 不能把插件路径一起改成另一套初始化契约

2. **插件版视为保护对象**
   - `src/entry-extension.ts` 默认不改
   - 除非执行阶段发现共享逻辑无法隔离，否则不允许主动修改插件入口

3. **不能只删日志**
   - 那只是掩盖问题
   - 重复触发链仍在
   - 真实设置读写行为仍可能不稳定

## 实施方案

### Step 1 — 修正 `tavern-storage.ts`，但只对 userscript/非 extension 路径生效

修改 `src/data/storage/tavern-storage.ts`，目标不是全局重构，而是**在 userscript 路径上**增加受控状态管理：

- 为 userscript 路径增加 bridge 初始化状态/已报告状态缓存，例如：
  - 未初始化
  - 初始化中
  - 暂时不可用
  - 已降级且已告警
  - 已恢复可用
- `getTavernSettingsNamespace_ACU()` 在 userscript 下需要：
  - 区分“bridge 暂时未就绪”和“bridge 已确认不可用”
  - 避免同一原因在启动窗口内无限重复 warn
  - 若 bridge 稍后恢复，应允许重新切回 Tavern 存储
- 插件路径下维持现有行为，不主动改变其获取 `extensionSettings` 的方式

### Step 2 — 改造 `getConfigStorage_ACU()` 的 userscript 降级策略

目标：不要让 userscript 启动期的每次同步配置访问都把同一个空 root 放大成一次新 warn。

具体要求：

- 在 userscript 路径中：
  - bridge 初始化未完成时，稳定回退到 IndexedDB / localStorage / memory
  - 只在状态变化时告警，而不是每次调用都告警
  - 如果后续 bridge 可用，允许恢复 Tavern 存储
- 在插件路径中：
  - 保持当前 `_isTavern` 语义和读取策略不变，除非执行阶段证明确有必要做兼容修补

### Step 3 — 调整 `loadSettings_ACU()` 的 userscript 首次加载顺序

修改 `src/service/settings/settings-service.ts`：

- 不要在 userscript 首次加载时，`void initTavernSettingsBridge_ACU()` 后立刻连续触发两次迁移和一次配置读取
- 应只对 userscript 路径增加首轮初始化保护，例如：
  - 首次加载先触发 bridge 尝试
  - 然后按受控顺序执行迁移与读取
  - 避免在 bridge 尚未准备好时就密集打到 `getConfigStorage_ACU()`
- 插件路径的启动顺序默认不改

这里的重点不是把 `loadSettings_ACU()` 整体异步重写给所有模式共用，而是**只修 userscript 这条出问题的启动路径**。

### Step 4 — 将 `src/index.ts` 纳入 userscript 启动检查范围

`src/index.ts` 是 JS 版入口，document ready 后直接 `mainInitialize_ACU()`。

执行阶段需要确认：

- 是否需要在 userscript 入口侧补充一次极轻量的 bridge 预热/保护
- 还是只在 `settings-service.ts` 内部处理即可

默认优先方案是：**先在共享存储层与 settings-service 层隔离修复**；只有这两层无法稳定解决时，才考虑对 `src/index.ts` 做最小入口调整。

### Step 5 — 插件入口只做回归检查，不作为默认修改目标

`src/entry-extension.ts` 从原计划中的“修改目标”降级为“回归验证对象”：

- 检查 userscript 修复后插件版是否仍正常
- 确认 `waitForTavernHelper()` 行为没有被共享模块的变更间接影响
- **不主动修改插件入口逻辑**

这一步不是偷懒，而是边界控制。用户已经明确告诉你插件没问题，继续往插件入口上动刀就是方向错误。

### Step 6 — 同步检查直接读取方的兼容性

执行阶段仍需要检查以下调用方：

- `src/data/repositories/profile-repo.ts`
- `src/service/template/template-preset-service.ts`
- `src/presentation/window/window-system.ts`
- `src/presentation/window/window-styles.ts`

目的不是改它们的业务逻辑，而是确认：

- userscript 路径下 `getConfigStorage_ACU()` 的去重/降级行为改变后，这些模块不会依赖旧的重复探测副作用
- 插件路径行为保持兼容

## 验证方案

### 单元/集成验证

优先运行：

- `tests/data/storage/tavern-storage.test.ts`
- `tests/service/settings/settings-service.test.ts`

必要时补充测试覆盖：

1. userscript 路径下 bridge 未就绪时，首次加载不会刷出多条相同 warn
2. userscript 路径下 bridge 稍后可用时，后续读取能恢复 Tavern 存储
3. userscript 路径下 bridge 永久不可用时，只执行受控降级，不重复刷屏
4. extension 路径行为不变，不被 userscript 修复误伤

### 构建与运行验证

#### 主验证：userscript / JS 版

根据用户实际使用的打包方式，重点验证：

- `npm run build` 产物（`dist/index.bundle.js`）
- 或用户实际运行的单文件 JS 版本

重点观察：

- 初始化阶段是否仍反复出现该 warn
- 设置是否仍能正常读取/保存
- Tavern bridge 不可用时是否正确降级
- bridge 稍后恢复时是否能恢复使用 Tavern 存储

#### 回归验证：插件版

- 运行 `npm run build:extension`
- 确认插件版没有新增初始化异常
- 确认不会因为共享模块改动而改变现有正常行为

## 风险与约束

1. `tavern-storage.ts` 是共享模块
   - 修改位置必须按环境分支精确收束
   - 否则 userscript 修复很容易误伤 plugin

2. `loadSettings_ACU()` 是多处调用的同步入口
   - 不能为了修 userscript，草率把所有模式都改成另一套异步契约
   - 否则插件版可能被连坐

3. `src/index.ts` 是否需要补入口保护目前仍是执行期判断项
   - 先修存储层与 settings-service
   - 不够再做最小入口补丁
   - 不能一上来就扩大改动面

## 实施完成标准

满足以下条件才算修好：

1. 打包后的 JS 版运行时，不再反复出现 `酒馆设置根对象不可用, 返回 null`
2. userscript 路径下，Tavern bridge 迟到/缺失时，系统只受控降级，不刷屏
3. userscript 路径下设置读写仍正常
4. 插件版行为保持不变，没有新增初始化问题
5. 相关测试通过，并完成 userscript 主验证 + extension 回归验证

## 备注

原计划里把 `entry-extension.ts` 也列为主动修改目标，这在用户补充“插件版没问题”之后已经不成立了。继续沿着那个方向改，只会把正常路径一起拖下水。

这次正确的修法不是“大一统重构”，而是：

> **把 userscript 专属问题隔离在 userscript 专属路径里修掉，同时把插件路径当成必须保护的稳定面。**

助手，这才叫基于事实做计划。之前如果不修正这个边界，计划写得再工整也只是把错误目标排版得更漂亮而已。
