# SQL 模式删除当前对话数据后重建运行时数据库实施计划

## 目标

修复一个已确认的问题，并保证插件源码版与打包后的 `index.js` 版行为一致：

1. **SQL 模式下删除当前对话的所有数据后，在删除本地/聊天快照数据之外，重建 SQLite 运行时数据库实例**，避免用户切换模板时继续沿用旧实例中的表结构或残留数据。

## 已确认事实

### 1. 删除流程当前缺少运行时数据库重建

- 删除保存主链在 `src/presentation/pages/visualizer-main-save.ts`。
- 当可视化编辑器中删除表格后，保存流程会调用：
  - `purgeSheetKeysFromChatHistoryHard_ACU(deletedKeysToPurge_ACU)`：从聊天记录中硬删除表数据快照。
  - `refreshMergedDataAndNotifyWithUI_ACU()`：刷新 UI / mergedData。
- **当前没有调用** `reloadStorageProvider()` 或 `disposeStorageProvider()`。
- 这意味着当前删除后，聊天记录和本地数据刷新了，但 SQLite 运行时数据库实例仍可能保留旧状态。

### 2. 项目已有可复用的运行时数据库重建能力

- `src/service/table/table-storage-strategy.ts` 已提供 `reloadStorageProvider()`。
- 该函数在 SQLite 模式下会：
  - 销毁当前 provider；
  - 重新创建 `SqlTableService`；
  - 执行 `loadFromChat()`，从最新聊天记录回灌数据。
- 所以本次不需要重新设计数据库生命周期，只需要把现有能力接到正确的删除时机上。

### 3. 打包产物中存在对应镜像逻辑

- `index.js` 中也存在：
  - 删除保存调用链；
  - `reloadStorageProvider()` 定义与调用位点。
- 所以实现时不能只改 `src/` 就算完成。
- 但 `index.js` 是构建产物还是仓库内手工维护文件，实施前仍需先确认，不能编造。

## 风险与未知项

### 已知风险

1. **重建时机错误会导致修复失效**
   - 必须在 `purgeSheetKeysFromChatHistoryHard_ACU()` 成功并且聊天记录已经被改写后，再执行 `reloadStorageProvider()`。
   - 如果提前重建，SQLite 仍会从旧聊天快照回灌，等于没修。

2. **删除失败时不能强行重建**
   - 如果硬删除失败，但仍继续重建运行时数据库，就会让内存实例和聊天快照继续不一致。
   - 只有在 purge 成功的前提下才允许重建。

3. **不能污染 native 模式**
   - 本次需求明确是 SQL/SQLite 模式。
   - native 模式下不应触发无意义的 SQLite 生命周期操作。

### 未确认事实（实施前必须核实）

1. 根级 `package.json` / 构建命令是否存在，以及 `index.js` 是否由 `src/` 构建生成。
2. 删除保存流程是否已有独立测试文件；如果没有，需要新增合适的测试文件，而不是硬塞进无关测试里。

## 实施方案

### 步骤 1：确认打包产物维护方式

先读取项目构建配置，确认以下事实：

- `index.js` 是否由 `src/` 构建生成；
- 如果是，使用什么命令生成；
- 如果不是，是否需要手工同步维护。

**目的**：避免出现“源码修了、打包版没变”或“手改打包文件却被构建覆盖”的返工。

---

### 步骤 2：修改删除流程，在 SQLite 模式下重建运行时数据库

修改文件：`src/presentation/pages/visualizer-main-save.ts`

#### 计划改动

在以下条件同时满足时触发重建：

1. `deletedKeysToPurge_ACU.length > 0`
2. `purgeSheetKeysFromChatHistoryHard_ACU()` 执行成功
3. 返回结果表明聊天记录已实际变更（例如 `r.changed === true`）
4. 当前模式是 SQLite

#### 插入时机

- 放在硬删除成功之后；
- 放在统一 `refreshMergedDataAndNotifyWithUI_ACU()` 之前。

#### 建议调用方式

- 调用 `reloadStorageProvider()`；
- 失败时：
  - 记录明确日志；
  - 给出用户可见 warning，或至少不要静默吞错；
  - 不要破坏原有保存主流程。

#### 预期效果

- 删除本地/聊天数据后，SQLite 运行时数据库同步重建；
- 用户切换模板时不会继续沿用旧实例中的旧表结构或残留数据。

---

### 步骤 3：为删除流程补自动化测试

目标文件：优先确认是否已有对应 save 测试；若无，则新增测试文件。

#### 至少覆盖以下场景

1. **SQLite 模式 + purge 成功 + 有实际变更** → 调用 `reloadStorageProvider()`。
2. **native 模式 + purge 成功** → 不调用 `reloadStorageProvider()`。
3. **purge 失败** → 不调用 `reloadStorageProvider()`。
4. **没有删除任何表 key** → 不调用 `reloadStorageProvider()`。

#### 验证重点

- 验证 provider 生命周期调用条件，而不是只验证 toast 文案。
- 如果测试只看提示文字，不看 `reloadStorageProvider()` 是否按条件触发，那种测试没有价值。

---

### 步骤 4：同步打包 JS 版

根据步骤 1 的结论二选一：

#### 情况 A：`index.js` 由构建生成

- 只修改 `src/`；
- 执行构建命令生成最新 `index.js`；
- 校验打包产物中已包含删除后调用 `reloadStorageProvider()` 的逻辑。

#### 情况 B：仓库要求直接维护 `index.js`

- 将删除流程修复同步到 `index.js` 对应代码段；
- 确保逻辑与 `src/` 保持一致，不允许插件版和打包版分叉。

## 验收标准

### 功能验收

在 SQLite 模式下，点击“删除当前对话的所有数据”后：

1. 本地数据被清理；
2. 聊天记录中对应表快照被硬删除；
3. SQLite 运行时数据库实例被重建；
4. 用户切换模板时不会残留旧模板数据或旧表结构。

### 测试验收

- 删除保存流程相关测试通过；
- 覆盖 SQLite 模式成功重建、native 模式不触发、purge 失败不触发、无删除键不触发这几类场景；
- 若需构建，构建成功且产物包含修复。

## 实施顺序

按业务风险排序：

1. 先确认 `index.js` 维护方式；
2. 再修删除后 SQLite 实例不重建的问题；这是状态一致性 bug，优先级最高；
3. 再补删除流程测试；
4. 最后同步打包产物。

## 备注

当前计划已经按用户要求收缩，只保留“删除当前对话所有数据后重建 SQLite 运行时数据库实例”这一项。唯一明确保留的未确认项是：**`index.js` 的生成方式**。实施时必须先核对构建配置，再决定是“改源码 + 构建”还是“源码/打包文件双改”。这不是细枝末节，而是交付是否完整的前置条件。
