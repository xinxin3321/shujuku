## 五、根因分析

### 5.1 不是三层架构，只是按文件夹分了类

当前代码只是把一个 10000+ 行的 IIFE 拆成了 `data/`、`service/`、`presentation/` 三个目录，但**函数之间的调用关系完全没有改**。旧代码里一个闭包内的函数可以随便互调，拆成模块后这些互调变成了跨层 import，暴露出了根本不存在的层次隔离。

### 5.2 核心设计错误：service 层在驱动 UI

当前的 service 层函数内部直接调用 `showToastr()`、`resetManualUpdateButton()`、`updateCardUpdateStatusDisplay()` 等 UI 操作。这不是"缺个抽象层"的问题——**是调用方向根本搞反了**。

正确的三层架构中，service 层永远不应该主动触发任何 UI 行为（无论直接还是间接）。它只负责：
- 接收参数
- 执行业务逻辑
- 返回结果（数据 / 状态 / 异常）

所有 UI 反馈（toast、按钮状态、进度条）都是 **presentation 层拿到 service 返回值之后自己做的事**。

```
❌ 当前（错误）：
presentation 调用 service.processUpdates()
  → service 内部: showToastr("更新成功")           // service 主动操作 UI
  → service 内部: resetManualUpdateButton()        // service 主动操作 UI
  → service 内部: updateCardUpdateStatusDisplay()  // service 主动操作 UI

✅ 正确：
presentation:
  const result = await service.processUpdates(params)
  if (result.success) {
    showToastr("更新成功")                          // presentation 自己决定
    resetManualUpdateButton()                       // presentation 自己决定
    updateCardUpdateStatusDisplay(result.statusText) // presentation 自己决定
  } else {
    showToastr("error", result.error)
  }
```

### 5.3 核心问题清单

| 问题 | 具体表现 | 根因 |
|------|----------|------|
| **service 层主动操作 UI** | 12 个 service 文件直接 import toast、按钮控制、状态显示 | service 函数没有"返回结果"的设计，而是在内部直接执行副作用 |
| **presentation 里放了业务逻辑** | `optimization-ui.ts` 有 `ensurePlotTasksCompat`、`normalizePlotPresetSelectionValue` 等纯计算函数 | 拆文件时按"原来在哪个 UI 附近"分类，而不是按职责分类 |
| **data 层里放了业务编排** | `table-repo.ts` 调用 `applyTemplateScopeForCurrentChat`、`refreshMergedDataAndNotify` | repo 里塞了"加载并初始化"的编排逻辑，应该只做纯 CRUD |
| **data 层读全局状态** | 7 个 data 文件 import `state-manager` 读 `settings_ACU` | data 函数没有参数化，直接从全局单例读配置 |
| **state-manager 混合了 UI 状态和业务状态** | `$popupInstance` jQuery 对象和 `settings` 业务配置放在同一个模块 | 没有区分"谁拥有什么状态" |

### 5.4 最重度耦合文件 TOP 5

| 文件 | 违规 import 数 | 角色 |
|------|---------------|------|
| `service/runtime/api-registry.ts` | 11 | 公共 API 注册表，直接引用 presentation 的 pages/components/triggers |
| `service/runtime/helpers-remaining.ts` | 7 | 旧 IIFE 的"剩余函数堆"，什么都有 |
| `service/runtime/init.ts` | 8 | 初始化逻辑，直接绑定 UI 事件 |
| `data/repositories/table-repo.ts` | 6 | 数据仓库，但包含业务编排逻辑 |
| `service/table/update-process.ts` | 6 | 更新流程，直接操作 UI 按钮状态 |

## 六、修复原则

### 核心原则：单向数据流 `presentation → service → data`

```
┌─────────────────┐
│  presentation   │  拥有: UI 状态、DOM 操作、用户交互、toast 通知
│  (调用方/编排者) │  职责: 调用 service，根据返回值决定 UI 行为
└────────┬────────┘
         │ 调用（传参数、拿结果）
         ▼
┌─────────────────┐
│    service      │  拥有: 业务状态（settings）、业务逻辑、计算
│  (纯逻辑层)     │  职责: 接收参数、执行逻辑、返回结果。绝不触发 UI。
└────────┬────────┘
         │ 调用（传参数、拿结果）
         ▼
┌─────────────────┐
│     data        │  拥有: 存储抽象（IndexedDB、localStorage、宿主 API）
│  (纯存取层)     │  职责: CRUD。不读全局状态，所有数据通过参数传入。
└─────────────────┘

┌─────────────────┐
│    shared       │  拥有: 常量、工具函数、类型定义、宿主 API 引用
│  (被所有层引用)  │  职责: 无状态的公共工具
└─────────────────┘
```

### 6.1 service 层改造：纯逻辑 + 返回值

**所有 service 函数必须遵守：**
- 不 import presentation 下的任何模块
- 不调用 toast、不操作按钮、不更新状态显示
- 通过返回值（`{ success, data, error, statusText }`）传达结果
- 需要进度回调时，接收 `onProgress?: (msg: string) => void` 参数，由调用方传入 UI 更新函数

**示例改造 — `update-process.ts`：**
```typescript
// ❌ 现在
async function processUpdates_ACU(...) {
  showToastr('info', '开始更新...');           // 直接操作 UI
  resetManualUpdateButton();                   // 直接操作 UI
  // ... 业务逻辑 ...
  updateCardUpdateStatusDisplay('完成');        // 直接操作 UI
}

// ✅ 改后
interface UpdateResult {
  success: boolean;
  updatedCount: number;
  errors: string[];
  statusText: string;
}
async function processUpdates(params, onProgress?: (msg: string) => void): Promise<UpdateResult> {
  onProgress?.('开始更新...');
  // ... 纯业务逻辑 ...
  return { success: true, updatedCount: 5, errors: [], statusText: '完成' };
}
```

### 6.2 data 层改造：参数化 + 纯 CRUD

**所有 data 函数必须遵守：**
- 不 import service 或 presentation 下的任何模块
- 不读 `state-manager` 的全局状态
- 所有需要的配置通过参数传入
- 只做数据的读写、序列化、反序列化

**示例改造 — `table-repo.ts`：**
```typescript
// ❌ 现在
import { settings_ACU } from '../../service/runtime/state-manager';
function loadTable() {
  const key = settings_ACU.isolationKey;  // 直接读全局状态
  // ...
}

// ✅ 改后
function loadTable(isolationKey: string) {  // 通过参数传入
  // ...
}
```

### 6.3 presentation 层改造：成为唯一的编排者

presentation 层负责：
1. 用户交互 → 收集参数
2. 调用 service 函数（传参数）
3. 拿到返回值 → 决定 UI 行为（toast、按钮状态、显示更新）
4. 持有所有 UI 状态（`$xxx` jQuery 对象）

**它是唯一允许同时引用 service 和操作 DOM 的层。**

### 6.4 state-manager 拆分

| 当前位置 | 内容 | 应归属 |
|----------|------|--------|
| state-manager | `$popupInstance`、`$statusMessageSpan` 等 50+ jQuery 对象 | `presentation/state/ui-refs.ts` |
| state-manager | `settings_ACU`、`currentJsonTableData` 等业务数据 | `service/state/app-state.ts` |
| state-manager | `SillyTavern_API`、`jQuery_API`、`toastr_API` 等宿主引用 | `shared/host-api.ts` |
| state-manager | `coreApisAreReady`、`allChatMessages` 等运行时标志 | `service/state/runtime-flags.ts` |

| 当前位置 | 函数 | 应归属 | 原因 |
|----------|------|--------|------|
| `presentation/components/optimization-ui.ts` | `ensurePlotTasksCompat`, `normalizePlotPresetSelectionValue`, `findPlotPresetByName`, 等 17 个纯逻辑函数 | `service/plot/plot-logic.ts` | 纯计算，不操作 DOM |
| `presentation/components/template-preset-ui.ts` | `getTemplatePreset`, `upsertTemplatePreset`, `getDefaultTemplateSnapshot` | `service/template/template-service.ts` | 数据操作，不操作 DOM |

### 6.6 验收标准

改造完成后，以下 import 规则必须 100% 通过：
