## 原则说明

1. 这段代码到底在做什么、为什么会在这里？

### A-1. service 层 20+ 次 showToastr 调用 — 严重程度：**高**

| `content-optimization.ts` | 1 |

**这个问题的本质是什么？** 解耦的核心原则：**service 层执行业务逻辑，返回结果。presentation 层拿到结果后自己决定怎么展示给用户。** service 层不应该主动感知 UI 的存在——不管它是弹 toast、刷新列表还是改变按钮状态。

`showToastr` 就是 service 层在说"我知道上面有个 UI，我要主动往上面弹通知"。这和 `$popupInstance_ACU.find()` 是同一类问题——只是一个操作 DOM，一个操作 toast 通知层。`showToastr` 放在 `service/runtime/toast-service.ts` 看起来路径合法，但**它做的事情不合法**——service 层通过它直接操纵了 UI。

**"service 层永远运行在浏览器环境所以可以弹 toast"是偷懒的借口。** 解耦的意义不是"能不能在 Node.js 跑"，而是**每一层只做自己该做的事、不越权**。service 层弹 toast = 越权。

**该怎么改：**

service 层函数不再调用 `showToastr`，而是返回结构化的结果：

```typescript
// 改之前（service 层越权）
export function switchIsolationProfile_ACU(code: string) {
    // ...业务逻辑...
    if (error) {
        showToastr_ACU('error', '切换标识代码失败');
        return;
    }
    showToastr_ACU('success', `已切换到标识代码: ${code}`);
}

// 改之后（service 层只返回结果）
export function switchIsolationProfile_ACU(code: string): { success: boolean; message: string } {
    // ...业务逻辑...
    if (error) {
        return { success: false, message: '切换标识代码失败' };
    }
    return { success: true, message: `已切换到标识代码: ${code}` };
}

// presentation 层调用方自己决定 UI 行为
const result = switchIsolationProfile_ACU(code);
if (result.success) {
    showToastr_ACU('success', result.message);
} else {
    showToastr_ACU('error', result.message);
}
```

每个文件需要逐函数改造，改造量按函数数统计：
- `helpers-remaining.ts`：9 处，部分在深层嵌套中需要改返回值链
- `merge-logic.ts`：4 处
- `settings-service.ts`：3 处
- 其余 4 个文件各 1 处

同样的原则适用于 `refreshMergedDataAndNotify` 等所有"service 层主动触发 UI 刷新"的调用。
